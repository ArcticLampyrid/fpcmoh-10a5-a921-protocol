# FPC1260 `10a5:a921` — protocol notes and capture demo

Reverse-engineered notes and a minimal working demo for the **FPC1260** USB
fingerprint sensor (USB ID `10a5:a921`). The demo opens the device, brings up
the TLS-PSK tunnel the sensor expects, and writes raw fingerprint images to
disk in a loop.

## Background

I first analyzed the **FPC1022** (`10a5:9800`) sensor on my own machine — those
notes live in the [`10a5:9800`
repo](https://github.com/ArcticLampyrid/fpcmoh-10a5-9800-protocol). When a
friend went looking for Linux support for the **FPC1260** (`10a5:a921`), it
caught my interest, and this is the result.

## How `10a5:a921` differs from `10a5:9800`

The two sensors share a command vocabulary but the `10a5:a921` path diverges in
several places — the demo and notes call these out:

- Bulk IN endpoint `0x81` (9800: `0x82`).
- Bulk event headers are **little-endian** (9800: big-endian).
- **The host is the TLS client**, the device is the server (9800: reversed).
- The 32-byte PSK is **generated by the host and pushed** to the device in a
  119-byte `SET_TLS_KEY` blob (9800: the host reads a sealed blob *from* the
  device). The blob construction is fully documented in the protocol notes.
- The raw frame is **64×176** and is rotated **90° CCW** into a 176×64 image.

## Tested hardware

Everything here was verified against a single unit reporting:

- USB ID: `10a5:a921`
- Sensor: `0x04ec`
- HW ID:  `0x0331`
- Image:  `64x176` raw (rendered `176x64` after rotation), 8-bit grayscale
- Firmware: `024.26.0.39`

Other `10a5:a921` revisions are likely close but untested. Field widths and offsets
in the protocol notes come from this unit only.

## Requirements

- Linux with libusb. The demo uses `pyusb`, which needs read/write access to
  the device (a udev rule or running as root).
- **Python 3.13+** — needed for `ssl.SSLContext.set_psk_client_callback`.
- OpenSSL with PSK ciphers available at `@SECLEVEL=0` (default on most
  distros).
- Python packages:
  - `pyusb`
  - `cryptography`
  - `Pillow` (only if you want PNG output)

Example udev rule (drop into `/etc/udev/rules.d/70-fpc1260.rules`, then
`sudo udevadm control --reload && sudo udevadm trigger`):

```udev
SUBSYSTEM=="usb", ATTRS{idVendor}=="10a5", ATTRS{idProduct}=="a921", TAG+="uaccess"
```

Make sure no other driver (`libfprint`/`fprintd`) is currently holding the
device.

## Usage

```sh
pip install pyusb cryptography Pillow
# Or on Arch Linux:
# sudo pacman -S python-pyusb python-cryptography python-pillow
python3 fpc1260-capture-loop.py --out captures --format png
```

Options:

```
--out PATH               output directory (default: fpc1260-captures)
--format {pgm,png,raw}   image format (default: pgm)
-n, --count N            stop after N captures (default: loop until Ctrl+C)
```

`pgm` is the most convenient default: no extra dependency, every image viewer
opens it. `raw` is just the `64*176 = 11264` grayscale bytes with no header.

Touch the sensor when prompted. Press `Ctrl+C` to stop; the demo issues the
`ABORT` + `SX sleep` cleanup so the next run starts cleanly.

## What's documented

The protocol notes cover what `fpc1260-capture-loop.py` actually exercises:

- The 12-byte little-endian bulk event header and the events used by the demo
  (`INIT_RESULT`, `TLS_RECORD`, `FINGER_DOWN`, `FINGER_UP`, `IMG`).
- The vendor control-transfer command table (`INIT`, `ARM`, `ABORT`,
  `TLS_INIT`, `TLS_DATA`, `INDICATE_S_STATE`, `GET_IMG`, `SET_TLS_KEY`,
  `CLR_IMG`).
- The full open + TLS handshake + capture + close sequence.
- The `INIT_RESULT` fields the demo reads (sensor, hardware id, firmware,
  image dimensions).
- TLS-PSK parameters: TLS 1.2, host-is-client, the offered cipher list,
  identity `"Disum PSK"`, ≤ 56-byte host→device fragments.
- Layout of the 119-byte `SET_TLS_KEY` blob the host pushes, and the fully
  deterministic HMAC-SHA256 / AES-256-CBC sealing (key/IV derivation, encode
  and decode flow, and a reproducible test vector).
- Layout of the TLS-plaintext image event after `GET_IMG`, and the observed
  image-transfer packet shape.

## Out of scope

- Enroll / verify / identify. In the community fork these run **host-side**
  (OpenCV feature extract + match), not on the device — the reader is used
  purely as a raw image sensor.
- The wider command surface defined in the driver but never issued on the
  capture path.
